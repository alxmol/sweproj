# Mini-EDR Test Document


## 1. Testing Strategy

### 1.1 Overall Strategy

Mini-EDR is verified at four testing levels: unit, integration, system, and regression. Each level targets a different scope of behavior and is owned by a specific role. Together they provide coverage from individual Rust functions up to full daemon operation against a live Linux kernel.

#### Unit Testing

Each of the seven crates (`mini-edr-sensor`, `mini-edr-pipeline`, `mini-edr-detection`, `mini-edr-tui`, `mini-edr-web`, `mini-edr-common`, `mini-edr-daemon`) maintains its own test module under `#[cfg(test)]`. Pure functions (feature computation, JSON serialization, threshold comparison) are tested directly; stateful components (`WindowAggregator`, `ModelManager`, `ProcReader`) are tested against mocks generated by the `mockall` crate. The subsystem engineer who owns a crate is responsible for its unit tests. Every pull request that modifies a crate must include corresponding test updates before it can be merged.

#### Integration Testing

Integration tests exercise the public interface between adjacent pipeline stages. Three integration seams receive primary attention:

- sensor -> pipeline (exercised by `TC-03` and `TC-07`)
- pipeline -> detection (exercised by `TC-17`)
- detection -> alerting (exercised by `TC-18`)

Integration tests use pre-recorded event traces rather than live eBPF probes so they run on any developer machine without elevated privileges. Ethan owns integration testing as the DevOps lead with support from the affected crate owner.

#### System Testing

System tests exercise the complete daemon binary inside a privileged Docker container running Ubuntu 24.04 with kernel 6.x. The daemon loads real eBPF probes, runs real inference, and emits real alerts while synthetic workloads execute in the same container. Four malicious-behavior fixtures are used: a reverse-shell simulator, a privilege-escalation attempt via `setuid exec`, a cryptominer CPU pattern emulator, and a TCP port scanner. Benign baseline workloads (compile job, `nginx`, idle desktop) are also exercised to verify absence of false positives (`TC-67`). The system test suite runs on a self-hosted GitHub Actions runner with `CAP_BPF` and `CAP_PERFMON` granted. Ethan also handles the system-test triage.

#### Regression Testing

Every bug fix merged into `main` contributes a regression test encoding the failing scenario and the corrected behavior. The regression suite also includes the four malicious-behavior scenarios required by `NFR-R04` (covered by `TC-44`), which must never produce false negatives on any commit to `main`. Regression runs on every push and every pull request via GitHub Actions; a regression failure is treated as a release blocker and auto-assigns the offending PR author.

### 1.2 Test Selection

Different testing levels use different selection techniques, matching the amount of internal visibility each level has into the system under test.

#### White-Box at the Unit Level

Unit tests are selected to exercise every non-trivial branch using statement and branch coverage: each reachable statement must be executed by at least one test, and each `if` / `else` / `match` arm must be taken in at least one direction. Path coverage is applied specifically to the alert-generation path in `AlertGenerator::evaluate`, where the interaction between threshold, pass-through mode, and feature-importance extraction produces multiple distinct paths.

#### Grey-Box at the Integration Level

Integration tests drive inputs through the public crate interface but allow inspection of a small number of internal probe points. For example, the sensor-to-pipeline test drives raw bytes into the `RingBufferConsumer` and then inspects the downstream `tokio::sync::mpsc` receiver to confirm the expected `SyscallEvent` sequence, providing stronger guarantees than pure black-box testing without coupling tests to private implementation details.

#### Black-Box at the System Level

System tests are derived entirely from the SRS and SDD without reference to implementation details. Four classical black-box techniques are used:

- **Equivalence partitioning:** Applied to the threat-score domain; the display partitions from `FR-T02` are tested in `TC-28`, and the behavioral alert partition defined by the threshold in `FR-D03` is tested in `TC-18`.
- **Boundary value analysis:** Follows at partition edges; `TC-19` covers the critical behavioral boundary at the threshold itself, and `TC-52` covers the validity boundaries of configuration values.
- **Decision tables:** Cover the alert-generation combinations of score-above-threshold, pass-through mode, and log level.
- **State transition tests:** Exercise the daemon lifecycle from the SDD state diagram: `TC-15` covers `Initializing -> Running`; `TC-16` covers `Initializing -> Degraded`; `TC-21` covers `Running -> Reloading -> Running` (successful swap); `TC-22` covers `Running -> Reloading -> Running` (retained old model on failure); `TC-47` covers any -> `Shutting Down`.

#### Use Case Coverage

Each of the eleven use cases `UC-01` through `UC-11` has coverage for its normal scenario and every alternate or exceptional scenario described in the SRS. Use-case scenarios that are already implicitly exercised by a functional-requirement test are noted in that test's Notes column; the remaining scenarios are covered by dedicated test cases in Section 2.13.

### 1.3 Adequacy Criterion

Test adequacy is measured quantitatively at each level. A build that falls below the criterion is rejected by CI and cannot merge without explicit waiver from the team as a whole.

#### Unit Level

Line coverage is measured per crate using `cargo-llvm-cov`. The baseline required by `NFR-M03` is 70 percent line coverage on the userspace daemon. The two security-critical crates, `mini-edr-sensor` and `mini-edr-detection`, are held to a higher internal target of 85 percent because a defect in either can either miss a real threat (detection) or introduce a kernel-safety problem (sensor). Current status is tracked in `TC-55`.

#### Integration Level

Every public function on a crate's inter-crate interface must be exercised by at least one integration test. This is tracked manually in the crate owner's checklist and verified at sprint review.

#### System Level

System-test adequacy is defined by three complementary criteria:

1. All 34 functional requirements `FR-S01` through `FR-W06` are covered by at least one test case; traceability is maintained in the Notes column of each row in Section 2 and summarized in each subsection intro.
2. All 28 non-functional requirements `NFR-P01` through `NFR-PO04` are covered by at least one test case or explicitly marked as covered via cross-reference to a consolidated test.
3. All 11 use cases `UC-01` through `UC-11` have their normal scenario plus every alternate and exceptional scenario enumerated in the SRS tested.

Every requirement and every use-case scenario is traced to its primary test case.

#### Performance

The seven performance requirements `NFR-P01` through `NFR-P07` each have a dedicated criterion benchmark or an explicit cross-reference: `NFR-P05` is covered by `TC-27` and `TC-32`; `NFR-P06` is covered by `TC-36`. A benchmark that regresses by more than 10 percent against the rolling 7-day baseline fails CI.

#### Adversarial

Beyond SRS-mandated coverage, Section 2.14 adds eight adversarial and negative-space tests (`TC-67` through `TC-74`) targeting failure modes not explicitly called out in the SRS: false-positive rate, deserializer robustness, path traversal, PID wraparound, deep ancestry, malformed model input, model classification accuracy, and fuzz testing. These are marked `[NEW]` in the Notes column. Two of them (`TC-67` false-positive rate and `TC-73` model accuracy) are recommended for promotion to formal NFRs in a future SRS revision because they measure properties that directly affect operator trust and detection utility.

### 1.4 Bug Tracking

GitHub Issues is the single source of truth for defects and enhancement requests. The team uses a structured label taxonomy, a required issue template, and a weekly triage meeting to keep the backlog current.

#### Label Taxonomy

- **Type:** `bug`, `enhancement`, `question`, `tech-debt`, `docs`
- **Priority:** `p0` (release blocker), `p1` (must fix before release), `p2` (should fix), `p3` (nice to have)
- **Subsystem:** `sensor`, `pipeline`, `detection`, `tui`, `web`, `daemon`, `build`
- **Status:** `needs-repro`, `ready`, `in-progress`, `blocked`, `fixed`, `wontfix`

#### Issue Template

A GitHub issue template enforces that every new bug report includes:

1. steps to reproduce
2. expected versus actual behavior
3. kernel version from `uname -r` and Linux distribution
4. daemon log output at debug level
5. the daemon's configuration file with secrets redacted

An issue missing any of these fields is labeled `needs-repro` and is not triaged until the submitter provides the missing information.

#### Workflow

Once a bug is confirmed by Ethan or the relevant subsystem owner, it is assigned to the owner of the affected crate. The fix is developed on a branch whose name encodes the issue number (for example, `fix/123-ring-buffer-overrun`), and the pull request title includes `Fixes #123` so GitHub automatically closes the issue on merge.

### 1.5 Technology and Tools

The Mini-EDR test toolchain is entirely Rust-native where possible.

#### Unit and Integration

- `cargo test` ŌĆö built-in Rust test harness; runs `#[test]` functions and integration tests in `tests/` directories.
- `cargo-nextest` ŌĆö faster parallel test runner for local development; preserves the `cargo test` interface so CI and local results are identical.
- `mockall` ŌĆö generates mock trait implementations, used to mock `InferenceModel` and `ProcReader` so unit tests don't require a real model file or real `/proc`.
- `proptest` ŌĆö property-based testing for feature-vector computation (`TC-12`), used to generate randomized event sequences and verify invariants.
- `cargo-fuzz` ŌĆö coverage-guided fuzzing, used for `TC-74` on `RingBufferConsumer::deserialize`.

#### Coverage

- `cargo-llvm-cov` ŌĆö produces HTML reports for local inspection and LCOV reports for CI upload.
- `Codecov` ŌĆö aggregates coverage across crates and enforces the regression threshold.

#### Performance and Profiling

- `criterion.rs` ŌĆö microbenchmarks for inference latency, feature-extraction throughput, JSON serialization.
- `flamegraph` and `perf` ŌĆö interactive profiling when a benchmark regression is flagged.
- `bpftool prog profile` ŌĆö kernel-side profiling of eBPF programs themselves, used for `NFR-P01` (`TC-39`).

#### System and End-to-End

- Docker with `--privileged` and `--cap-add=BPF,PERFMON` for containerized system tests; base image pinned via `sha256` digest.
- `testcontainers-rs` ŌĆö programmatic container lifecycle from within Rust integration tests.
- Custom malicious-behavior simulators under `tests/fixtures/malware/` (used by `TC-44`) and benign baselines under `tests/fixtures/benign/` (used by `TC-67`).
- `curl`, `websocat`, and `fantoccini` (Rust WebDriver binding) for exercising the axum HTTP endpoints and WebSocket/SSE stream.

#### Static Analysis and Hygiene

- `cargo clippy --all-targets --all-features -- -D warnings` on every PR; warnings are errors.
- `cargo fmt --check` to enforce formatting.
- `cargo audit` for known CVEs in dependencies.
- `cargo deny` for license and source-policy enforcement.

#### Continuous Integration

CI runs on GitHub Actions across two runner classes. Default runners (`ubuntu-latest`, kernel 6.x) execute unit tests, integration tests, clippy, fmt, audit, and coverage. A self-hosted runner with `CAP_BPF` and `CAP_PERFMON` executes the system-test suite and eBPF-specific benchmarks. A nightly matrix job boots a KVM VM pinned to kernel 5.8 to verify CO-RE portability (covered by `TC-04` and `TC-57`).

## 2. Test Cases and Test Results

This section enumerates the 74 test cases that together constitute the Mini-EDR test suite. The suite covers all 34 functional requirements, all 28 non-functional requirements, and all 11 use cases (normal and alternate/exceptional scenarios) from the SRS, plus 8 adversarial and negative-space tests added during test planning.

### Conventions

- Sections are ordered by SRS requirement group: functional requirements first (`FR-S`, `FR-P`, `FR-D`, `FR-A`, `FR-T`, `FR-W`), then non-functional requirements (`NFR-P`, `NFR-R`, `NFR-AV`, `NFR-SE`, `NFR-M`, `NFR-PO`), then use-case scenarios not already covered, then adversarial tests.
- Each requirement has exactly one primary test case. When a single test covers more than one requirement, every covered requirement is listed in that test's Notes column, and the additional requirements are marked as cross-referenced in the section intro of their originating group. No requirement is tested twice, and no requirement is omitted.
- Tests covering scenarios added beyond the SRS (adversarial and negative-space cases in Section 2.14) are tagged `[NEW]` in the Notes column.
- Status legend: **Pass**, **Pending**, **Fail**.

### SRS Addendum: FR-D03 Ambiguity (resolved during test planning)

**FR-D03 interpretation decision ŌĆö SRS revision recommended**

The SRS wording ŌĆ£the system shall generate an alert when the threshold is exceededŌĆØ is ambiguous between strict `>` and `>=`. Without disambiguation, the boundary case (score exactly equal to threshold) is untestable: either interpretation yields a different expected result.

After team review, the implementation adopts `>=` (score `>= threshold` triggers an alert). This matches standard industry practice for threshold-based alerting, makes the default threshold value `0.7` unambiguously meaningful, and matches how operators will reason about ŌĆ£set the threshold to 0.7ŌĆØ statements.

Test cases `TC-18` (score above threshold) and `TC-19` (score exactly at threshold) both follow this interpretation. Since the SRS and SDD are already completed and turned in, this change is recorded here in the testing document.
